# niri Desktop Environment — Agent Instructions

## What This Is
A Wayland desktop environment for the niri compositor. Every component is a standalone Rust binary using GTK4 for UI. No Electron, no Python, no web stack.

## Programs
| Directory | Binary | Purpose |
|-----------|--------|---------|
| `niri-shell/` | `niri-shell` | Core shell / compositor interface |
| `niri-dock/` | `niri-dock` | Dock bar (active + pinned apps) |
| `niri-panel/` | `niri-panel` | Status bar (workspaces, media, stats, notifications) |
| `niri-launcher/` | `niri-launcher` | App launcher (fuzzy search, calculator) |

Each is an **independent** Cargo binary crate with its own `Cargo.toml`. Do not add cross-crate dependencies between them.

## Build Commands
```sh
cargo check --quiet          # fast syntax/type check
cargo build                  # compile
cargo test                   # run tests
cargo clippy -- -D warnings  # lint (must be clean)
cargo fmt                    # format
```
Always run `cargo check --quiet` before committing. A commit that fails `cargo check` is a broken commit.

## File Layout Per Program
Every program uses **exactly** this source layout:
```
<program>/
  Cargo.toml
  src/
    main.rs            ← calls <component>_ui::run(), nothing else
    <component>_ui.rs  ← ALL GTK4 code, zero business logic
    <component>_backend.rs ← ALL state/logic, zero GTK4 imports
```
This separation is a hard constraint. Never mix UI and backend in the same file.

## Non-Negotiable Rules

### NO Adwaita — ever
- Never import `libadwaita`, `adw`, or any `adw::` type.
- Never use Adwaita CSS classes (`"suggested-action"`, `"card"`, `"boxed-list"`, etc.).
- Use raw `gtk4::` widgets and custom CSS only.
- Never add `libadwaita` to any `Cargo.toml`.

### UI file rules (`*_ui.rs`)
- Only `gtk4`, `glib`, `gio` imports.
- Import state/types from `crate::<component>_backend` — never duplicate logic.
- Load CSS via `gtk4::CssProvider` with inline string or a bundled file.
- All interactive widgets must have `gtk4::Accessible` roles set.

### Backend file rules (`*_backend.rs`)
- Zero `gtk4` / `glib` / `gio` imports.
- Pure Rust: structs, enums, trait impls, `std` only.
- All public functions return `Result<T, E>` with typed errors.

### Error handling
- All errors use `thiserror`. No `Box<dyn Error>` in public APIs.
- No `.unwrap()` or `.expect()` outside `#[cfg(test)]` and `main()`.
- Use `?` for propagation. Document invariants with a comment if `expect` is truly necessary.

### Logging
- Use `log::info!`, `log::warn!`, `log::error!` etc.
- Never use `println!` or `eprintln!` in production code paths.
- `env_logger` is the runtime backend — already in all `Cargo.toml`s.

### Dependencies
1. Check `std` first — use `OnceLock`, `mpsc`, `Path`, etc. before adding a crate.
2. Prefer crates already used in the workspace (`log`, `env_logger`, `thiserror`, `gtk4`, `glib`).
3. Only add a new crate if it covers a real protocol/algorithm that would take >30 lines to reimplement correctly.
4. Never add: `once_cell`, `lazy_static`, `itertools`, `dirs`, `strum`, `derive_more`.

## GTK4 Versions
```toml
gtk4 = { version = "0.9", features = ["v4_12"] }
glib = "0.20"
log = "0.4"
env_logger = "0.11"
thiserror = "2"
```

## Code Quality
- Edition: `2021`. Never use `2024`.
- Max line length: 100 characters (`rustfmt`).
- Error type names: `<Subsystem>Error` (e.g., `DockError`, `PanelError`).
- No global mutable state.
- No `#[allow(warnings)]` or `#[allow(clippy::all)]` at crate root.
- Dead code goes in `doa/` or gets deleted — never left commented out.
